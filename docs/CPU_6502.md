# 2A03 CPU Milestone

The `cpu-6502` crate implements the documented NMOS 6502 instruction set as
used by the NTSC Ricoh RP2A03, with decimal arithmetic disabled while the D flag
remains readable and writable.

## Implemented with generated and independent single-step tests

- The canonical set of 151 documented opcode encodings returns explicit
  mnemonic, addressing mode, base cycles, instruction length, and page-cross
  policy.
- A committed 190-vector architectural sample curated from the MIT-licensed
  `SingleStepTests/65x02` RP2A03 suite checks all 151 documented encodings. It
  compares final PC, SP, A, X, Y, status, declared RAM, and instruction cycles;
  it also includes both cycle profiles for all 23 page-penalty encodings and
  2/3/4-cycle profiles for all eight branches.
- The identity-checked Kevin Horton `nestest` V1.00 trace passes all 8,991 rows
  and 8,990 transitions through mapper 0. The supported set includes the exact
  76 stable undocumented encodings exercised by that fixture: NOP aliases,
  LAX, SAX, DCP, ISC, SLO, RLA, SRE, RRA, and the `$EB` SBC alias.
- Three project-owned NROM diagnostics match 47-row architectural traces
  generated by pinned BSD-3-Clause py65 through the real parser, mapper-0 bus,
  runner, and CLI. They cover CPU RAM mirrors, PRG mapping/RAM, ignored ROM
  writes, trainer endpoint preload, stack and control flow, branches, and
  page-cross cycles; reset and bus order are not checked.
- Instruction-level architectural state and cumulative cycle traces.
- Immediate, zero-page, indexed, absolute, indirect, indexed-indirect,
  indirect-indexed, accumulator, implied, and relative addressing.
- Wrapping zero-page pointers and the NMOS `JMP ($xxFF)` high-byte wrap quirk.
- Binary ADC/SBC carry, borrow, overflow, zero, and negative behavior.
- Stack order and architectural effects for JSR/RTS, BRK/RTI, PHP/PLP, and
  PHA/PLA.
- Branch taken and page-cross cycle increments.
- Reset-vector power-on state with seven recorded cycles.
- Failure-atomic exhaustion at the final seven cycle-counter values. At an
  instruction boundary, `clock` reserves the supported eight-cycle maximum
  before touching the bus, so it may stop conservatively even when a shorter
  instruction would numerically fit.

## Interrupts and reset (live cycle granularity)

Interrupt recognition and entry share the same one-bus-access `clock` boundary
as instructions. The implementation was compared numerically with the pinned
external `perfect6502` transistor-level oracle; the upstream simulator and
netlist stay outside the repository and are not runtime dependencies.

- Line inputs: `set_nmi_line` is edge-triggered and latches a pending NMI until
  it is serviced or the CPU is reset; `set_irq_line` is level-triggered and is
  re-evaluated at every poll.
- The physical interrupt poll occurs before each instruction's final cycle.
  Variable branch paths poll after the opcode fetch when not taken, after the
  offset read when taken on the same page, and after the taken-address dummy
  read when crossing a page. `pending_interrupt` exposes the accepted result.
  NMI outranks IRQ, and IRQ is accepted only when the sampled `I` flag is clear.
- One-instruction `I`-flag delay: `CLI`, `SEI`, and `PLP` change the flag after
  the interrupt poll, so the poll observes the pre-instruction value; every other
  instruction, including `RTI`, is observed with its post-instruction value.
- An accepted IRQ or NMI automatically begins on the next `clock` call. Its
  seven calls issue two reads from the current PC, push PC high, PC low, and
  status (`B` clear and bit 5 set), then read the low and high vector bytes.
  `begin_interrupt` exposes the same live sequence for controlled drivers;
  `service_interrupt` remains a compatibility wrapper that drains it.
- NMI can hijack an in-progress BRK or IRQ through the low-PC push. The selected
  vector is locked after that cycle: an earlier NMI uses `$FFFA`, while a later
  edge leaves the IRQ/BRK vector at `$FFFE` and remains pending.
- `begin_reset` schedules a warm reset whose seven calls issue two current-PC
  reads, three stack reads while decrementing `SP`, and the two `$FFFC` vector
  reads. It sets `I`, drops a latched NMI, and preserves `A`, `X`, `Y`, and the
  other flags. `reset` is the compatibility wrapper that drains the sequence.
- The seven-cycle interrupt and reset counts are reserved before any bus access,
  so an exhausted cycle counter leaves the CPU and bus unchanged.

## Per-cycle bus accuracy

Every sampled documented instruction encoding emits the 6502's expected
per-cycle bus access sequence for its pinned case, verified against the
SingleStepTests oracle: each vector in `singlestep_vectors.rs` carries its
ordered `bus_cycles` (address, value, read/write), and a strict test requires the
CPU's recorded bus trace to match all 190 vectors byte for byte. The coverage
was reached incrementally (74 -> 96 -> 108 -> 166 -> 190). It covers:

- the opcode fetch and operand/pointer reads;
- the dummy read of the byte after a one-byte implied or accumulator opcode;
- the zero-page-indexed and indexed-indirect base dummy reads;
- the un-fixed-address dummy read of absolute-indexed and indirect-indexed
  accesses, issued on a page cross for reads and always for writes and
  read-modify-writes;
- the read-modify-write write-back of the original value before the modified
  value; and
- the stack, control-flow, branch, and BRK per-cycle sequences.

`clock` performs exactly one live bus read or write on each successful call and
returns `InProgress`, `InstructionComplete(StepTrace)`,
`InterruptComplete(InterruptEntry)`, `ResetComplete(ResetEntry)`, or an
`UnsupportedOpcode` outcome after consuming the failed opcode-fetch cycle. The
existing `step` API is a compatibility wrapper that repeatedly clocks the same
instruction. All 227 currently decoded encodings use the live cycle state
machine; the strict 190-vector test checks the documented sample after every
individual call.

`core-nes::NesMachine::clock` composes this interface with mapper 0 and the NTSC
scheduler. Every successful CPU bus cycle advances exactly 12 master ticks and
three PPU dots, reports a bus fault on the cycle that caused it, and exposes any
timed PPU event for that cycle. Scheduler-overflow validation happens before the
CPU access so the CPU and scheduler cannot diverge on that error path.

## Explicitly not yet claimed

- An asynchronous reset pin that can take over in the middle of an instruction.
  The verified warm-reset sequence is explicitly scheduled at a machine
  boundary with `begin_reset`.
- DMA stalls and RDY behavior. The machine scheduler advances live instruction,
  interrupt, and reset cycles, but those arbitration paths do not exist yet.
- Undocumented opcodes outside the exact 76-encoding `nestest` set, including
  jam and hardware-sensitive unstable encodings.
- Per-cycle bus order for the 76 stable undocumented encodings. Their accepted
  `nestest` trace verifies instruction-boundary state and cycle totals, not
  ordered reads and writes.
- Exhaustive input-state coverage. The pinned single-step sample is independent
  evidence across every documented encoding, not all 10,000 upstream scenarios
  per opcode.

Those limitations prevent this crate or machine boundary from being called a
complete cycle-accurate NES CPU. It is a live cycle-stepped CPU layer with
independent sampled instruction evidence, independently measured interrupt and
reset entry, and full mapper-trace evidence. After exact identity verification, the strict CLI
allows writes to `$4004`-`$4007` and `$4015`; the reviewed log makes exactly
five terminal writes there. This is not APU emulation, and the normal mapper bus
still faults unimplemented I/O.
